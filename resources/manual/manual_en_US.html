📖 1. System Overview

This system is a practical outcome of the "Comprehensive Programming Course Design" at the School of Cyber Science and Engineering, Huazhong University of Science and Technology. It integrates theoretical knowledge of computer networks, data structures, algorithm design, and C++/Python programming into a real‑world network security analysis scenario. Facing increasingly complex cyber attacks, real‑time monitoring and intelligent analysis of massive traffic have become critical to maintaining stable network operation. This system closely meets actual needs, efficiently processes real network session data, automatically builds network topology models, and integrates a series of functions such as traffic statistics, behavior analysis, anomaly detection, and result visualization, providing network administrators and security analysts with an intuitive and efficient all‑in‑one analysis tool.

🖥️ 3. Main Interface Layout

The main window is divided into five major areas:

  • ① Menu Bar: File operations, thread count adjustment, language switching, help documentation, etc.
  • ② Data File Bar: Shows the path of the currently loaded data file; the “Browse” button switches data sources.
  • ③ Function Tabs: Traffic ranking, path finding, anomaly detection, custom rules, subgraph visualization.
  • ④ Output Panel: Contains three sub‑pages – “Run Log”, “Data Table”, and “Path & Details” – displaying analysis results in real time.

  • If you select a JSON or HTML file, the program enters “View‑Only Mode”. You can browse the existing graph structure or export it to other formats, but you cannot perform new analysis operations.
📌 Tip: Temporary files generated during runtime are stored in the system’s temporary directory by default. You can periodically clean them to free up space.

🔧 5. Core Features Explained

5.1 Traffic Ranking

Located in the first tab. Quickly discover the most active or suspicious nodes in the network.

  • Total Traffic Ranking: Calculates total inbound and outbound traffic for each IP node and sorts descending.
  • HTTPS Traffic Ranking: Filters sessions using HTTPS (transport protocol TCP with destination port 443), aggregates HTTPS traffic per node, and sorts descending.
  • Outbound Traffic Ratio: Filters nodes where outbound traffic accounts for more than a specified threshold (default 80%), sorts by total traffic descending, and displays the exact ratio.

After clicking “Execute Traffic Ranking”, the results appear in the “Data Table” sub‑page. If you need to export the detailed results, you can copy the corresponding text from the “Run Log”.

5.2 Path Finding

Specify a source IP and a destination IP; the system computes the optimal path based on the selected strategy.

  • Minimum Congestion Path: Uses DataSize / Duration as the congestion metric for each edge, finds the path with the lowest total congestion.
  • Minimum Hop Path: Finds the path with the fewest hops (shortest route).
  • Minimum Risk Path: Uses a custom cost function (considering port security and port scanning behavior) to find the path with the lowest total risk.
  • Compare Three Strategies: When this checkbox is selected, clicking the button computes optimal paths for all three strategies simultaneously and displays a comparison in the “Path & Details” sub‑page.

After clicking “Find Path”, the system automatically switches to the “Path & Details” sub‑page, showing the sequence of IPs along the path along with corresponding congestion/risk values and other details.

5.3 Anomaly Detection

This tab contains three sub‑pages: Port Scan, DDoS Target, and Star Structure.

📡 Port Scan Detection

If a node initiates many connections to the same neighbor IP using many different destination ports, and its outbound traffic accounts for more than a threshold of its total traffic, it is flagged as a potential scanner. Default outbound ratio threshold is 80%, and port count threshold is 20. You can adjust these values to control sensitivity.

💥 DDoS Target Detection

If a node receives connection requests from a large number of distinct neighbors and its inbound traffic exceeds a threshold, it is flagged as a possible DDoS target. Default neighbor count threshold is 20, inbound traffic threshold is 1GB. Both can be adjusted.

⭐ Star Structure Detection

If a node communicates with many leaf nodes (nodes that connect only to it), it is marked as a star structure center. Default leaf node threshold is 20, adjustable.

Clicking the “Detect” button on any sub‑page displays the results in the “Data Table” and automatically generates a subgraph that highlights the anomalous nodes and their connections.

5.4 Custom Rules

This feature allows users to define access control rules based on IP ranges, protocols, ports, and traffic thresholds, and then find all communication records that violate the rules.

  • Target IP: The central node to monitor.
  • IP Range: Can be specified in CIDR format (e.g., 192.168.1.0/24) or as a start‑end IP range.
  • Optional Parameters: Protocol number, source port, destination port, maximum traffic threshold.
  • Rule Type: “Deny” means the target IP is forbidden to communicate with IPs in the range and cannot use the specified protocol/ports; “Allow” means the target IP can only communicate with IPs in the range and only using the specified protocol/ports.

After clicking “Check Violations”, the “Data Table” sub‑page lists all violating communication records, including IP addresses, protocol, ports, and the reason for violation. If you need to export the detailed results, you can copy the corresponding text from the “Run Log”.

5.5 Subgraph Visualization

Enter any IP address and click “Generate Subgraph”. The system finds the connected component containing that node and renders an interactive network topology using the PyVis engine, displayed in the right‑hand canvas. You can:

  • Drag nodes and zoom with the mouse wheel.
  • Hover over nodes to see IP addresses; hover over edges to see traffic information.
  • Use the “Export” menu to save the current subgraph as a JSON or HTML file.

❓ 9. Frequently Asked Questions (FAQ)

Q: The application seems unresponsive after loading a PCAP file.
A: Large PCAP files may take considerable time to parse and convert. Please be patient.
Q: “No nodes in the graph” when generating a subgraph.
A: Possible reasons: ① The entered IP address does not exist in the current topology; ② There are no other nodes communicating with that IP; ③ The CSV file does not contain valid IP entries. Verify that the node exists first.
Q: Large network topologies load slowly.
A: Due to performance limitations of PyVis and Qt WebEngine, we recommend loading data and then entering a specific IP to narrow down the subgraph instead of rendering the entire graph.
Q: Custom rule check returns no results.
A: Check whether the target IP is correct, and whether the IP range, protocol, ports, etc., are too restrictive. Try relaxing the conditions gradually.
Q: How is the risk defined for the “Minimum Risk Path”?
A: Risk is evaluated based on a “risk cost” assigned to each edge. The cost consists of two parts:
  • Base security factor: 1.0 if the communication uses secure protocols (e.g., HTTPS, SSH); otherwise 10.0.
  • Port scan penalty factor: Calculated from the number of distinct port pairs seen on that edge. More port pairs yield a higher penalty (max 3.0).
The final edge risk cost = base security factor × port scan penalty factor. The system finds the path from source to destination with the smallest total risk cost, which effectively avoids potential port‑scanning behaviors and low‑security links.
Q: How can I get software updates?
A: This is a course design project; version 1.0 is the final release. No update schedule is planned.

📋 10. Appendix: Data Format Reference

The standard input CSV file must contain the following columns, encoded in UTF‑8:

Source,Destination,Protocol,SrcPort,DstPort,DataSize,Duration
192.168.1.10,10.0.0.22,6,54321,443,1200,0.5
192.168.1.15,10.0.0.22,6,12345,80,800,0.3
...

Column Descriptions

  • Source / Destination: Dotted‑decimal IPv4 addresses.
  • Protocol: IP protocol number (e.g., 6=TCP, 17=UDP, 1=ICMP).
  • SrcPort / DstPort: Transport layer ports (leave empty if the protocol does not use ports).
  • DataSize: Total bytes transferred in the session.
  • Duration: Session duration in seconds.
📌 Tip: You can use Wireshark’s “Conversation Statistics” feature to export CSV data, or directly use raw .pcap files – the system automatically performs flow aggregation and preprocessing.
